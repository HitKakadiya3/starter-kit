## Commands

All commands run from `typestest/` (the app root).

- `npm run dev` — start Vite dev server on `http://localhost:8080` (host `::`, HMR overlay disabled).
- `npm run build` — production build. `npm run build:dev` builds in development mode (keeps `lovable-tagger` plugin).
- `npm run lint` — ESLint across the repo (TypeScript + React Hooks + React Refresh rules).
- `npm run test` — Vitest single run (jsdom, `src/test/setup.ts` auto-loaded).
- `npm run test:watch` — Vitest watch mode.
- Run a single test file: `npx vitest run src/path/to/file.test.ts`.
- Run tests matching a name: `npx vitest run -t "partial test name"`.
- Playwright: config comes from `lovable-agent-playwright-config`. Import test utilities from `./playwright-fixture.ts`, not `@playwright/test` directly.

Package manager: both `bun.lock`/`bun.lockb` and `package-lock.json` are committed. `npm` works; prefer not to switch lockfiles unless asked.

## Architecture

This is a **Vite + React 18 + TypeScript SPA** originally generated by Lovable. It implements a 16-type personality quiz (MBTI-inspired) with a funnel-style multi-page flow.

### Quiz data flow (the core domain)

The scoring pipeline is small and pure — understand it before touching anything under `src/utils/` or `src/hooks/`:

1. **`src/utils/questions.ts`** — flat list of `Question` objects, each tagged with a `dim` (`E/I | S/N | T/F | J/P`) and `pole` (which letter the statement leans toward).
2. **`src/hooks/useQuiz.ts`** — iterates `questions`, calls `applyAnswer(scores, pole, weight)` per response. Weights come from `ScaleSelector` in `{-2, -1, 0, 1, 2}`. Negative weights add to the opposite pole (see `opposites` map in `src/utils/scoring.ts`).
3. On the final answer, `useQuiz` flips `isComplete`. `QuizPage` reacts by navigating to `/calculating` and **passes `scores` via `location.state`** — there is no global store. Pages downstream of `/quiz` read `location.state.scores`; navigating to them directly bounces back to `/`.
4. **`src/utils/scoring.ts`** — `calculateType(scores)` picks the larger side of each dimension to form the 4-letter type; `calculatePercentages` produces per-trait bar data.
5. **`src/utils/types.ts`** — `typeData` is a `Record<string, TypeData>` with copy for all 16 types. **`src/hooks/useResults.ts`** memoises `type` + `typeData[type]` + percentages for the results/report pages.

When adding a new question, it becomes part of the score denominator automatically — no other changes needed. When adding a new type field, update the `TypeData` interface and every entry in `typeData`.

### Routing & page funnel

`src/App.tsx` is the only router. Only `IntroPage` and `NotFound` are statically imported; every other route is `React.lazy` — lazy modules must have a default export. Routes form a funnel: `/` → `/instructions` → `/quiz` → `/calculating` → `/email` → `/checkout` → `/cross-sell` → `/details` → `/results` (aliased to `PremiumReportPage`). A `ScrollToTop` component resets scroll on every navigation. Notable aliasing: `<Route path="/results">` and `<Route path="/premium-report">` both render `PremiumReportPage`.

### UI stack

- **shadcn/ui** components live in `src/components/ui/` (configured in `components.json`, base color `slate`, no prefix). Treat these as vendored — edit freely, but prefer composing over forking.
- **Tailwind** tokens are HSL CSS vars defined in `src/index.css` and mapped in `tailwind.config.ts`. Custom token groups: `quiz-*` (progress, trait bar, badge) and `scale-*` (agree/disagree/neutral gradients used by `ScaleSelector`). Use these tokens rather than hard-coded colors so theming stays consistent.
- Font: Plus Jakarta Sans (via Tailwind `font-sans`).
- Toasters: both `@/components/ui/toaster` (Radix) and `sonner` are mounted in `App.tsx`. `useToast` in `src/hooks/use-toast.ts` is the Radix-based one.

### Path alias

`@/*` maps to `src/*` (set in `vite.config.ts`, `vitest.config.ts`, `tsconfig.json`). `react`, `react-dom`, JSX runtimes, and `@tanstack/react-query` are deduped in the Vite config to avoid duplicate copies.

### TypeScript config quirks

`tsconfig.json` disables `noImplicitAny`, `strictNullChecks`, `noUnusedLocals`, `noUnusedParameters`. ESLint also disables `@typescript-eslint/no-unused-vars`. Don't rely on these bailouts for new code, but don't fight them either when touching existing files.

### Testing

- Vitest with `jsdom`, globals enabled. Setup in `src/test/setup.ts` shims `window.matchMedia`.
- Test files: `src/**/*.{test,spec}.{ts,tsx}`.
- Use `@testing-library/react` + `@testing-library/jest-dom` for components; pure utils (`scoring.ts`, etc.) don't need a DOM.

### Lovable integration

- `lovable-tagger` runs only in dev builds (tags components for the Lovable editor) — don't remove the `mode === "development"` guard in `vite.config.ts`.
- `.lovable/plan.md` is Lovable's scratchpad; safe to ignore unless asked.
